1) Acesso e Leitura dos Dados em Formato XML

A primeira etapa consiste em desenvolver uma soluo para ler os dados do arquivo XML original. A abordagem escolhida foi a criao de um parser robusto em Python, utilizando a biblioteca padro xml.etree.ElementTree para processar a estrutura hierrquica do XML.

Arquitetura da Soluo

  1. data_class.py: Um mdulo contendo dataclasses que espelham a estrutura do XML. Esta abordagem oferece tipagem esttica, validao implcita de campos e uma forma limpa e orientada a objetos de representar os dados em memria.
  2. data_parser.py: Contm a classe HistoricoCreditoXMLParser, responsvel por mapear cada tag e atributo para as dataclasses correspondentes, tratando tipos de dados e campos opcionais.

Premissas Utilizadas

  • Estrutura do XML: O arquivo XML bem-formado e segue o schema definido no dicionrio de dados.
  • Tipos de Dados: Datas so tratadas como strings, valores monetrios como float e quantidades como int.
  • Tratamento de Erros: O parser resiliente a campos opcionais ausentes e registra warnings em caso de erros de converso de tipo, sem interromper a execuo.

2) Armazenamento em Formato Analtico

Com os dados em memria, o prximo passo armazen-los em um formato otimizado para anlises. A soluo principal proposta utiliza o Google BigQuery, mas uma abordagem relacional tradicional tambm uma alternativa vivel.

Soluo Principal: Modelo Desnormalizado (Google BigQuery)

  1. Plataforma: Google BigQuery: Soluo serverless e escalvel, ideal para queries analticas (OLAP). Seu suporte nativo a dados aninhados (STRUCT/RECORD) perfeito para representar a estrutura do XML sem perda de informao e sem a necessidade de JOINs complexos.

  2. Formato de Ingesto: NDJSON (Newline Delimited JSON): Formato padro para ingesto em batch no BigQuery. gerado facilmente a partir dos nossos objetos e permite parsing paralelo eficiente pela plataforma.

  3. Modelagem: Tabela nica com Campos Aninhados: Mantemos a hierarquia original em vez de achatar os dados. Isso melhora a performance de consultas (UNNEST mais otimizado que JOINs em larga escala) e torna o schema mais intuitivo.

def preparar_para_bigquery(dados: EnvioHistoricoCredito, output_filename: str) -> str:
    """Converte o objeto dataclass para um dicionrio e o salva em formato NDJSON."""
    dados_dict = asdict(dados)
    
    with open(output_filename, 'w', encoding='utf-8') as f:
        json_record = json.dumps(dados_dict, ensure_ascii=False)
        f.write(json_record + '\n')
        
    logging.info(f"Arquivo '{output_filename}' pronto para ingesto no BigQuery.")
    return output_filename

Alternativa: Modelo Relacional Normalizado (SQL Server / PostgreSQL)

Uma abordagem alternativa seria usar um banco de dados relacional tradicional. Nesse cenrio, "achatamos" a estrutura hierrquica do XML, distribuindo os dados em mltiplas tabelas normalizadas. Esta modelagem, conhecida como Esquema Estrela (Star Schema), clssica em Data Warehousing.

Modelo de Dados Relacional

O processo de ETL (Extrao, Transformao e Carga) seria mais complexo, pois precisaria popular diversas tabelas e manter a integridade referencial atravs de chaves primrias e estrangeiras. ndices seriam criados nas chaves estrangeiras (FK) e em colunas frequentemente usadas em filtros (WHERE) para otimizar a performance dos JOINs.

Vantagens e Desvantagens

  • Vantagens:
    • Integridade de Dados: Fortemente garantida por chaves estrangeiras.
    • Reduo de Redundncia: A normalizao evita a repetio de dados.
    • Familiaridade: Modelo amplamente conhecido por analistas e desenvolvedores SQL.
  • Desvantagens:
    • Complexidade nas Consultas: Requer mltiplos JOINs para reconstruir a viso completa, o que pode ser menos performtico para anlises complexas em grandes volumes.
    • ETL Mais Complexo: O processo de carga precisa coordenar a insero em vrias tabelas.

3) Melhorias na Documentao

Uma boa documentao crucial para a governana e utilizao correta dos dados. O dicionrio de dados poderia ser enriquecido com as seguintes informaes:

  1. Tipagem e Formato Explcito: Especificar o tipo de dado esperado para cada campo (ex: DtCtrc: DATE(YYYY-MM-DD), VlObgc: DECIMAL(18, 2)).

  2. Descrio de Domnios: Para campos de cdigo (CdMdld, TipCli), fornecer uma tabela "de-para" com o significado de cada valor (ex: CdMdld: 001 - Crdito Pessoal).

  3. Exemplos Prticos: Incluir um valor de exemplo para cada campo para elucidar o formato esperado (ex: NrCtr: "Ex: CTA-789123-01").

  4. Regras de Negcio: Explicitar as regras que governam o preenchimento (ex: "O campo DtContCtCsr s preenchido se CdMdld for Consrcio").

  5. Significado dos Comandos: Documentar o que cada valor para os campos Cmdo* significa (ex: I - Incluso, A - Alterao, E - Excluso).

  6. Metadados Gerais: Incluir controle de verso do layout, origem dos dados, frequncia de atualizao e contatos responsveis (Data Stewards).

Consulta 4a: Contratos e Soma de Valor por Modalidade

Lista todos os contratos nicos por CPF, sua modalidade, valor contratado e uma coluna adicional com a soma total contratada por aquele cliente naquela modalidade.

Abordagem BigQuery (Desnormalizada)

Utiliza UNNEST para acessar os dados aninhados de forma eficiente.

-- Sintaxe para Google BigQuery
SELECT
  cliente.idfc_cli AS cpf_cnpj_cliente,
  operacao.nr_ctr AS numero_contrato,
  operacao.cd_mdld AS modalidade_contrato,
  COALESCE(operacao.det_opr.vl_obgc, 0) AS valor_contratado,
  SUM(COALESCE(operacao.det_opr.vl_obgc, 0)) OVER (PARTITION BY cliente.idfc_cli, operacao.cd_mdld) AS soma_valor_por_modalidade_cliente
FROM
  `quod_dwh.credit_catalog.historico_credito_final`,
  UNNEST(Cli) AS cliente,
  UNNEST(cliente.operacoes) AS operacao
WHERE
  operacao.nr_ctr IS NOT NULL
ORDER BY
  cpf_cnpj_cliente,
  modalidade_contrato;

-- Sintaxe para SQL Server / PostgreSQL
WITH OperacoesCliente AS (
    SELECT
        c.idfc_cli,
        o.nr_ctr,
        o.cd_mdld,
        o.vl_obgc AS valor_contratado
    FROM
        DIM_CLIENTE c
    JOIN
        FACT_OPERACAO o ON c.id_cliente = o.id_cliente
)
SELECT
    idfc_cli AS cpf_cnpj_cliente,
    nr_ctr AS numero_contrato,
    cd_mdld AS modalidade_contrato,
    valor_contratado,
    SUM(valor_contratado) OVER (PARTITION BY idfc_cli, cd_mdld) AS soma_valor_por_modalidade_cliente
FROM
    OperacoesCliente
ORDER BY
    cpf_cnpj_cliente,
    modalidade_contrato;

Consulta 4b: Indicador de Parcelas em Atraso

Para cada contrato de cada cliente, gera um indicador booleano (TRUE/FALSE) que aponta se existe ao menos uma parcela anterior no quitada (em atraso).

-- Sintaxe para Google BigQuery
WITH ContratosComStatusDeAtraso AS (
  SELECT
    cliente.idfc_cli,
    operacao.nr_ctr,
    (
      SELECT LOGICAL_OR(p.dt_vnct_pcl_ant < CURRENT_DATE() AND NOT EXISTS (SELECT 1 FROM UNNEST(p.pgto_pcl_ant)))
      FROM UNNEST(operacao.pcl_ant) AS p
    ) AS tem_atraso_normal,
    (
      SELECT LOGICAL_OR(p_csr.dt_vnct_pcl_ant_csr < CURRENT_DATE() AND NOT EXISTS (SELECT 1 FROM UNNEST(p_csr.pgto_pcl_ant_csr)))
      FROM UNNEST(operacao.pcl_ant_csr) AS p_csr
    ) AS tem_atraso_consorcio
  FROM
    `quod_dwh.credit_catalog.historico_credito_final`,
    UNNEST(Cli) AS cliente,
    UNNEST(cliente.operacoes) AS operacao
)
SELECT
  idfc_cli AS cpf_cnpj_cliente,
  nr_ctr AS numero_contrato,
  COALESCE(tem_atraso_normal, FALSE) OR COALESCE(tem_atraso_consorcio, FALSE) AS indicador_possui_atraso
FROM
  ContratosComStatusDeAtraso
WHERE
  nr_ctr IS NOT NULL
ORDER BY
  cpf_cnpj_cliente,
  numero_contrato;

-- Sintaxe para SQL Server / PostgreSQL
SELECT
    c.idfc_cli AS cpf_cnpj_cliente,
    o.nr_ctr AS numero_contrato,
    -- Verifica se existe alguma parcela para a operao que esteja em atraso
    CAST(CASE 
        WHEN EXISTS (
            SELECT 1
            FROM FACT_PARCELA par
            WHERE 
                par.id_operacao = o.id_operacao
                AND par.dt_vnct < GETDATE() -- ou CURRENT_DATE para PostgreSQL
                AND NOT EXISTS (
                    SELECT 1 
                    FROM FACT_PAGAMENTO pag 
                    WHERE pag.id_parcela = par.id_parcela
                )
        ) THEN 1 
        ELSE 0 
    END AS bit) AS indicador_possui_atraso
FROM
    DIM_CLIENTE c
JOIN
    FACT_OPERACAO o ON c.id_cliente = o.id_cliente
ORDER BY
    c.idfc_cli,
    o.nr_ctr;

1) Preparao e Seleo de Features

O primeiro passo na modelagem sempre arrumar a casa. Nossos dados brutos precisam de um trato antes de poderem ser usados.

O que vamos fazer aqui :

  • Limpar o Lixo: Variveis com muitos dados faltando (mais de 70%) geralmente mais atrapalham do que ajudam. Vamos remov-las, junto com colunas de identificao como id e safra.
  • Tapar os Buracos: Nos dados que sobraram, ainda existem alguns valores faltando. Vamos preench-los usando a mediana, que uma forma segura de fazer isso sem ser muito afetado por valores extremos.
  • Escolher os Melhores Jogadores: Mesmo depois da limpeza, ainda temos muitas variveis. Para no confundir nosso modelo, vamos usar uma tcnica chamada RFE para nos ajudar a escolher as 50 variveis que parecem ter mais "poder de fogo" para prever o resultado.

def load_and_clean_csv(filepath):
    try:
        with open(filepath, 'r', encoding='utf-8') as f:
            lines = f.readlines()
        cleaned_lines = [line.strip().strip('"') for line in lines]
        csv_string = "\n".join(cleaned_lines)
        csv_io = io.StringIO(csv_string)
        df = pd.read_csv(csv_io)
        return df
    except FileNotFoundError:
        return pd.DataFrame()

df = load_and_clean_csv('Anexo 2 - Base Modelagem.csv')

if not df.empty:
    numeric_cols = df.select_dtypes(include=np.number).columns.tolist()
    df[numeric_cols] = df[numeric_cols].mask(df[numeric_cols] < 0, np.nan)

    missing_percentage = (df.isnull().sum() / len(df)) * 100
    high_missing_vars = missing_percentage[missing_percentage > 70]
    
    features_to_drop = ['id', 'safra'] + high_missing_vars.index.tolist()
    df_cleaned = df.drop(columns=features_to_drop, errors='ignore')
    
    X = df_cleaned.drop(columns='resposta')
    y = df_cleaned['resposta']

    preprocessing_pipeline = Pipeline([
        ('imputer', SimpleImputer(strategy='median')),
        ('scaler', StandardScaler())
    ])
    X_processed = preprocessing_pipeline.fit_transform(X, y)
    X_processed = pd.DataFrame(X_processed, columns=X.columns)
    
    NUM_FEATURES_TO_SELECT = 50
    estimator = LogisticRegression(solver='liblinear', class_weight='balanced')
    selector = RFE(estimator, n_features_to_select=NUM_FEATURES_TO_SELECT, step=10)
    
    selector = selector.fit(X_processed, y)
    
    selected_features = X.columns[selector.support_].tolist()
    X_final = X_processed[selected_features]
    print(f"{len(selected_features)} features selecionadas e prontas para a modelagem.")

2) Construo e Comparao de Modelos

Com nossos 50 melhores "ingredientes" selecionados, hora de cozinhar! Vamos testar algumas "receitas" (modelos) diferentes para ver qual delas produz o melhor resultado.

Nossos competidores so:

  • Logistic Ridge: Um modelo clssico, rpido e confivel. como o "arroz com feijo" da modelagem, mas muitas vezes funciona surpreendentemente bem.
  • Random Forest: Um modelo mais complexo que usa vrias "rvores de deciso" para chegar a um consenso. timo para pegar relaes mais complicadas nos dados.
  • XGBoost: O "campeo de competies". um algoritmo super poderoso, famoso por conseguir timos resultados em dados como os nossos.

Para ter certeza de que a competio justa, vamos usar uma tcnica chamada Validao Cruzada. Em vez de testar cada modelo s uma vez, vamos test-los 5 vezes em diferentes "fatias" dos dados e tirar a mdia. Isso nos d uma ideia muito mais robusta de quem o verdadeiro vencedor. A nota que daremos a cada um a AUC-ROC, uma mtrica que mede o quo bom o modelo em separar os "bons" dos "maus".

3) Otimizao de Hiperparmetros (Tuning)

J sabemos que o Logistic Ridge o melhor modelo para o nosso caso. Agora, vamos fazer um "tuning" nele. como se a gente j tivesse escolhido o melhor carro de corrida, e agora vamos ajustar os "botes" e "alavancas" (os hiperparmetros) dele para ver se conseguimos um desempenho ainda melhor.

Vamos focar no parmetro C, que controla a "fora" da regularizao do modelo.

O nosso modelo final, um Logistic Ridge com os parmetros otimizados, alcanou uma AUC-ROC de 0.66 no teste final. Isso significa que ele tem um poder de discriminao razovel, sendo consideravelmente melhor do que uma escolha aleatria.

Em vez de aprovar ou negar crdito automaticamente, o modelo poderia sinalizar os clientes que ele acha que so da Classe 1 para uma anlise mais detalhada por um especialista.
